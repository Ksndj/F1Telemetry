# F1Telemetry Agent Rules

本仓库内的自动化代理、编码助手和协作者在执行任何任务前，必须遵守以下规则。

## 强制要求

- 先阅读现有项目结构和命名风格，再开始修改。
- 尽量复用已有类型，不要重复造轮子。
- 只完成当前里程碑范围内的内容，不改无关模块。
- 不要一次性重构无关模块。
- 新增代码必须能通过构建。
- 禁止全量测试，用分模块顺序测试；模块测试顺序由执行者按由简到难原则决定。
- 保持 WPF + MVVM 结构清晰。
- 不要把业务逻辑写进 code-behind。
- 所有公开类和方法添加简洁注释。
- 需要新增依赖时，先选择最少依赖方案。
- 如果发现信息不足，先做最稳妥实现，并在最后明确标记待确认项。

## 版本控制约束

- 每个里程碑完成后必须执行 Git 提交与推送。
- 仅提交当前里程碑相关文件，禁止混入无关改动。
- 提交信息必须清晰描述本次里程碑目标与改动内容。
- 必须推送到远端分支；若无 upstream，则先设置 upstream。
- 严禁执行以下操作：
  - `git reset --hard`
  - `git push --force`
  - 改写历史（`rebase` / `amend` 已发布提交）
- 严禁提交以下内容：
  - 临时构建文件（`obj` / `bin`）
  - `wpftmp` / 临时 `csproj`
  - 本地数据库文件
  - API Key 或任何敏感信息
- 若 `push` 失败，只允许输出错误信息，不做破坏性操作。

## 强制输出格式

每次任务完成后，必须输出以下内容：

1. 修改了哪些文件
2. 为什么这样设计
3. 还有哪些风险

如任务涉及版本控制，还必须补充：

1. 当前分支名
2. 提交 hash
3. 提交信息
4. 推送目标（remote/branch）
5. 是否存在未提交文件

如任务本身包含额外交付项，允许在以上三项之外补充，但不得省略这三项。

## 模块边界

- 共享抽象、接口、基础模型优先放入 `F1Telemetry.Core`。
- 优先复用现有抽象、模型和工具类型，避免在相邻模块重复定义近似结构。
- WPF 界面行为优先通过 ViewModel、绑定和命令实现。
- 具体实现放入职责对应模块，不跨层混写。

## 规范来源

- 详细规则见 `TASK_SPEC.md`。
- 如 `AGENTS.md` 与普通说明性文档冲突，以本文件和 `TASK_SPEC.md` 为准。
