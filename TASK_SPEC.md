# 项目任务规范

本文件用于约束 F1Telemetry 项目后续迭代中的任务执行方式、修改边界和交付格式。
本规范与仓库根目录 `AGENTS.md` 共同构成强约束执行规则。

## 通用要求

- 先阅读现有项目结构和命名风格，再开始修改。
- 尽量复用已有类型，不要重复造轮子。
- 只完成当前里程碑范围内的内容，不改无关模块。
- 不要一次性重构无关模块。
- 新增代码必须能通过构建。
- 保持 WPF + MVVM 结构清晰。
- 不要把业务逻辑写进 code-behind。
- 所有公开类和方法添加简洁注释。
- 需要新增依赖时，先选择最少依赖方案。
- 如果发现信息不足，先做最稳妥实现，并在最后明确标记待确认项。

## 版本控制约束

- 每个里程碑完成后必须执行 Git 提交与推送。
- 仅提交当前里程碑相关文件，禁止混入无关改动。
- 提交信息必须清晰描述本次里程碑目标与改动内容。
- 必须推送到远端分支；若无 upstream，则先设置 upstream。
- 严禁执行以下操作：
  - `git reset --hard`
  - `git push --force`
  - 改写历史（`rebase` / `amend` 已发布提交）
- 严禁提交以下内容：
  - 临时构建文件（`obj` / `bin`）
  - `wpftmp` / 临时 `csproj`
  - 本地数据库文件
  - API Key 或任何敏感信息
- 若 `push` 失败，只允许输出错误信息，不做破坏性操作。

## 交付输出要求

每次任务完成后，统一输出以下内容：

1. 修改了哪些文件
2. 为什么这样设计
3. 还有哪些风险

如任务涉及版本控制，补充输出以下内容：

1. 当前分支名
2. 提交 hash
3. 提交信息
4. 推送目标（remote/branch）
5. 是否存在未提交文件

如当前任务另有明确要求，可在以上三项之外补充，但不得省略这三项。

## 适用范围

- `F1Telemetry.App`
- `F1Telemetry.Core`
- `F1Telemetry.Udp`
- `F1Telemetry.Analytics`
- `F1Telemetry.AI`
- `F1Telemetry.TTS`
- `F1Telemetry.Storage`
- `F1Telemetry.Tests`

## 执行说明

- 修改前优先确认当前任务与现有模块职责是否一致。
- 如需扩展 UI，优先通过 ViewModel、命令和绑定完成。
- 如需新增抽象，优先放入 `F1Telemetry.Core`，并优先复用既有抽象、模型与工具类型，避免重复定义。
- 如需新增实现，优先放入对应职责模块，避免跨层混杂。
- 如需补充公共 API，公开类和公开方法应添加简洁注释，说明职责或用途。
- 如信息不完整，先采用最稳妥、最小假设的实现路径，并在任务结尾单独标注待确认项。
- 如当前任务形成一个里程碑交付，完成后应只暂存并提交本次里程碑相关文件，避免把无关变更混入版本历史。
- 如需推送远端，优先确认当前分支与 upstream 关系，`push` 失败时只报告错误并停止，不进行破坏性 Git 操作。

## 强约束执行方式

- 仓库根目录 `AGENTS.md` 作为代理执行入口规范，要求默认遵守本文件内容。
- `Directory.Build.targets` 会在构建时校验 `TASK_SPEC.md` 存在，并输出强制规范提醒。
- 如后续增加自动化脚本、检查器或任务模板，应继续复用本文件中的同一套要求，避免出现多份互相冲突的规范。
