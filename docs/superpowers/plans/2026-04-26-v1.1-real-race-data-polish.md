# F1Telemetry V1.1 Real Race Data Polish Plan

> 本文件仅记录 V1.1 待办规划，不代表本次立即开发。当前任务不改业务代码、不改版本号、不构建、不打包。

## 背景

V1.0.2 已完成 UI 架构、页面拆分、侧边栏、图标、安装包、关闭进程修复。实际跑了 4 圈排位赛后，软件能收到 UDP 数据，但真实比赛数据展示仍需要围绕可读性、赛制语义、AI 分析播报、事件摘要和 AI/TTS 实战体验继续打磨。

V1.1 的目标不是推翻 V1 架构，而是在 V1.0.2 可用基础上，让软件从“能收到并显示数据”进一步变成“比赛中看得懂、用得上、不干扰驾驶”的工程师助手。

## 范围约束

- 不重构现有 UDP / Analytics / AI / TTS / Storage 主链路。
- 不改变 V1.0.2 发布包、安装方式或版本号。
- 不引入自动更新、后台检查或额外发布体系。
- 不把排位赛、冲刺赛、正赛混成同一套提示语义。
- 不用空白、`-` 或内部编码替代用户可理解的比赛状态。
- 不再把实时图表作为驾驶中核心交互；速度、输入、燃油和胎磨趋势优先转成 AI/TTS 短播报。

## V1.1 待办清单

### 1. 赛道名称中文化

- [ ] `trackId` 不直接显示为“赛道 ID x”。
- [ ] 建立 `trackId -> 中文简称` 映射。
- [ ] 顶部状态栏、Overview、AI 输入统一使用中文赛道名。
- [ ] 未识别赛道显示“未知赛道（ID x）”，避免只暴露裸编号。

### 2. 轮胎类型中文化

- [ ] 不显示 `A16` / `L16` 等内部编码作为正式 UI 文案。
- [ ] 显示红胎 / 黄胎 / 白胎 / 半雨胎 / 全雨胎。
- [ ] 对手列表、胎况摘要、AI 输入统一使用可读名称。
- [ ] `telemetry restricted` 显示“遥测受限”，不要误显示为未知轮胎。
- [ ] 原始 compound 编码只放 Tooltip 或调试模式。

### 3. 赛制类型区分

- [ ] 练习赛 / 排位赛 / 冲刺排位 / 冲刺赛 / 正赛中文显示。
- [ ] 不同赛制页面重点不同。
- [ ] AI Prompt 带 `sessionType`，避免排位赛给正赛策略建议。
- [ ] 排位赛强调有效圈、交通、ERS 准备和轮胎准备。
- [ ] 正赛强调进站、胎龄、油耗、前后车差距和策略风险。

### 4. 实时趋势数据转 AI 播报

- [ ] 将当前圈速度、油门 / 刹车摘要写入 AI 输入。
- [ ] 将多圈燃油趋势摘要写入 AI 输入。
- [ ] 将多圈四轮磨损趋势摘要写入 AI 输入。
- [ ] AI/TTS 只播短结论，不要求驾驶中盯实时图表。
- [ ] 数据不足时在 AI/TTS 日志说明等待原因，例如“等待本圈采样”“完成一圈后生成趋势”“等待轮胎磨损数据”。

### 5. Overview 事件摘要压缩

- [ ] Overview 只显示最近几条关键事件。
- [ ] 单条事件限制长度，超长省略。
- [ ] 完整内容放事件日志页。
- [ ] 摘要优先保留进站、安全车、黄旗、无效圈、低油、高胎磨、攻击/防守窗口和关键 AI 建议。

### 6. 对手列表可读化

- [ ] 位置、轮胎、胎龄、进站状态、差距可读化。
- [ ] `telemetry restricted` 显示“遥测受限”。
- [ ] 内部编码放 Tooltip 或调试模式。
- [ ] 进站状态显示为“赛道上 / 进站中 / 维修区 / 已进站 x 次”等可读文案。
- [ ] 前车 / 后车摘要优先展示对驾驶决策有用的信息。

### 7. 日志分类增强

- [ ] 增加 System / UDP / RaceEvent / AI / TTS / Storage 分类。
- [ ] 原始 UDP 事件不要刷屏干扰比赛阅读。
- [ ] 原始 `BUTN` / `SPTP` / `SEND` 等事件默认降噪，必要时放入调试模式或折叠区。
- [ ] 比赛事件使用可读文本，例如黄旗、安全车、进站、最快圈、圈无效。

### 8. 顶部状态栏信息重排

- [ ] 比赛时优先展示连接状态、赛道、赛制、圈数、天气。
- [ ] UDP PPS、监听端口、开始/停止监听作为次级信息。
- [ ] 端口输入和监听按钮在比赛中不应抢视觉焦点。
- [ ] 小窗口下关键比赛信息优先保留，调试/配置类信息可折叠或弱化。

### 9. 空数据状态优化

- [ ] AI 播报、单圈、胎况、对手页没有数据时显示明确原因。
- [ ] 示例：“等待完成第一圈”。
- [ ] 示例：“等待 CarDamage 包”。
- [ ] 示例：“等待 SessionHistory 包”。
- [ ] 示例：“对手遥测受限”。
- [ ] 不要只显示 `-` 或大面积空白。
- [ ] 区分真实缺包、游戏限制、遥测受限和功能尚未接线。

### 10. 排位赛专用提示

- [ ] 显示当前有效圈 / 无效圈。
- [ ] 显示最快圈、上一圈、本节剩余时间。
- [ ] 提醒交通风险、放空窗口、ERS 准备、轮胎准备。
- [ ] AI 在排位赛中不要给进站窗口、undercut、overcut 或长距离策略类建议。
- [ ] TTS 优先播报无效圈、交通风险、ERS 准备和放空窗口。

### 11. 正赛专用提示

- [ ] 显示胎龄、进站次数、前后车差距、油耗余量、ERS 可用量。
- [ ] 增加 undercut / overcut 风险提示。
- [ ] 根据前后车轮胎、胎龄、进站状态判断攻击/防守窗口。
- [ ] AI 在正赛中重点给策略、轮胎、燃油、交通建议。
- [ ] TTS 优先播报进站窗口、低油、高胎磨和前后车风险。

### 12. AI/TTS 真实比赛联调

- [ ] AI 输入使用中文赛道、赛制、轮胎、关键事件。
- [ ] AI 输入区分练习赛、排位赛、冲刺赛、正赛。
- [ ] TTS 只播关键结论，减少刷屏。
- [ ] 限制 AI/TTS 每圈播报频率，避免覆盖黄旗、进站、低油等高优先级事件。
- [ ] 失败时给出可见提示：网络失败、API Key 未配置、JSON 解析失败、限流等。

## 建议拆分

### V1.1-M1：显示语义中文化

范围：赛道名、赛制、轮胎类型、对手列表基础可读化。

### V1.1-M2：AI 播报输入与空状态

范围：实时趋势数据转 AI 输入、AI/TTS 日志等待原因、单圈/胎况/对手页空数据说明。

### V1.1-M3：事件摘要与日志分类

范围：Overview 事件摘要压缩、事件日志承载完整内容、日志分类和原始 UDP 事件降噪。

### V1.1-M4：赛制专用提示

范围：练习赛、排位赛、冲刺排位、冲刺赛、正赛不同重点的 UI 和提示逻辑。

### V1.1-M5：AI/TTS 真实比赛联调

范围：AI Prompt 语义增强、TTS 频控、关键事件优先级、错误可见化和真实比赛联调。

## V1.1 验收方向

- 真实进入练习赛、排位赛、冲刺赛、正赛时，顶部状态栏显示中文赛道和中文赛制。
- 轮胎显示为红胎 / 黄胎 / 白胎 / 半雨胎 / 全雨胎，对手遥测受限时明确提示。
- 实时趋势数据能进入 AI 分析播报；数据不足时说明等待原因。
- Overview 事件摘要短而关键，完整事件保留在事件日志页。
- 对手列表能快速读懂轮胎、胎龄、进站状态和前后差距。
- 排位赛不出现正赛进站窗口或长距离策略建议。
- 正赛能围绕轮胎、燃油、前后车、进站窗口给出策略提示。
- AI/TTS 输出短、准、符合赛制，并且不会覆盖高优先级比赛事件。

## V1.1.1 测试升级项：真实数据验证与 UDP 记录

V1.1.1 用于在 V1.1-M5 完成后，通过真实 F1 25 完整正赛 UDP 数据校准显示、AI 趋势播报、轮胎、赛制、事件摘要和 AI/TTS 输入质量。本阶段重点是记录、验收和分析，不默认开启调试采集能力。

### 目标

- 增加真实赛道和真实比赛数据验证能力。
- 支持可选 UDP raw log 记录，用于后续映射、AI 趋势播报、事件检测和离线 replay/parser 调试。
- 用完整正赛数据校准 AI 趋势播报、轮胎映射、赛制识别、进站状态、LapSummary 稳定性和 AI/TTS 输入质量。
- 在实测确认前，actual dry compound `16-22` 继续使用安全兜底：`未知胎（编码 x）`。

### 1. AI 趋势播报真实数据验收

- [ ] 刚进赛道后，当前圈速度、油门 / 刹车采样进入 AI 输入摘要。
- [ ] 刚进赛道但未完成圈时，AI/TTS 日志说明 `完成至少一圈后生成燃油趋势`。
- [ ] 刚进赛道但未收到胎磨数据时，AI/TTS 日志说明 `等待轮胎磨损数据`。
- [ ] 跑完 1 圈后，燃油趋势进入 AI 输入摘要。
- [ ] 跑完 1 圈后，单圈历史同步出现。
- [ ] 跑完 2 圈后，胎磨趋势有数据则进入 AI 输入摘要。
- [ ] 跑完 2 圈后，胎磨无数据时继续记录等待原因。
- [ ] 胎磨无数据时不允许输出误导性的 0 值结论。
- [ ] 重新开始一次 session 后，AI 趋势摘要不残留上一局数据。

### 2. 轮胎映射实测校准

- [ ] 用记录的 UDP 数据分析 visual compound。
- [ ] 用记录的 UDP 数据分析 actual compound。
- [ ] 用记录的 UDP 数据分析 tyre set allocation。
- [ ] 用记录的 UDP 数据分析 session type。
- [ ] 用记录的 UDP 数据分析 tyre age。
- [ ] 确认不同赛制下红胎 / 黄胎 / 白胎映射是否稳定。
- [ ] actual dry compound 与红黄白的对应关系未确认前，不把 `16-22` 硬映射为红胎 / 黄胎 / 白胎。

### 3. UDP 原始数据记录

- [ ] 新增可选 UDP raw log 记录功能。
- [ ] 默认关闭，仅用于调试和数据分析。
- [ ] 记录完整正赛 UDP 包，用作后续映射、AI 趋势播报和事件检测校准。
- [ ] 日志文件按时间和 session 命名。
- [ ] 不记录 API Key、设置文件、数据库内容或任何敏感配置。
- [ ] raw log 文件必须被 `.gitignore` 排除，禁止提交。

### 4. 正赛完整数据采集

- [ ] 用户跑一次完整正赛并开启 UDP raw log。
- [ ] 用完整记录分析轮胎映射。
- [ ] 用完整记录分析进站状态。
- [ ] 用完整记录分析 sessionType。
- [ ] 用完整记录分析 AI 趋势摘要采样完整性。
- [ ] 用完整记录分析 LapSummary 生成稳定性。
- [ ] 用完整记录分析 AI/TTS 输入质量。

### 5. 回放 / 离线分析准备

V1.1.1 不要求实现完整回放 UI，但 raw log 格式必须方便后续离线 replay/parser 调试。

建议每条 raw log 记录包含：

- timestamp
- packet type / packet id
- session uid
- parsed header 摘要
- raw bytes 或 base64

### 输出物

- [ ] 一份完整正赛 UDP raw log。
- [ ] 一份实测校准报告，记录 AI 趋势播报、轮胎映射、sessionType、进站状态和 AI/TTS 输入质量。
- [ ] 一组后续修复项清单。
